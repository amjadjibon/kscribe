# Manifests & install — single source of truth

The Helm chart (`charts/kscribe`) is the single source of truth for how kscribe is
installed. Everything else is generated from it. This doc says what to edit, what
not to, and how the pieces flow.

## The pipeline

```
Go //+kubebuilder markers ──controller-gen──▶ charts/kscribe/crds/*.yaml          (CRD schemas)
   (api/, internal/controller)   make manifests └─▶ charts/kscribe/templates/role.yaml (ClusterRole)

charts/kscribe ── values.yaml + templates ── the ONE thing you edit by hand
        │
        └── helm template --include-crds ──▶ deploy/kscribe.yaml                  (flat manifest)
                 scripts/build-manifest.sh
```

Two install paths, same source:

- **Helm:** `helm install kscribe ./charts/kscribe` — uses the chart directly.
- **kubectl:** `kubectl apply -f deploy/kscribe.yaml` — uses the rendered copy.

They can't drift, because `deploy/kscribe.yaml` is produced from the chart.

## What is generated (never hand-edit)

| File | Generated by | Source |
|------|--------------|--------|
| `charts/kscribe/crds/*.yaml` | `make manifests` (controller-gen) | `// +kubebuilder:object` / CRD markers in `api/v1alpha1` |
| `charts/kscribe/templates/role.yaml` | `make manifests` (controller-gen) | `// +kubebuilder:rbac` markers in `internal/controller/doc.go` |
| `deploy/kscribe.yaml` | `scripts/build-manifest.sh` (`helm template`) | the whole chart |

Each carries a generated-file marker or lives under a generator. Editing them by
hand is overwritten on the next `make manifests` / `make manifest-check`.

## What you edit by hand

There are exactly **two** intentional edit points:

1. **Go code** — for anything about the API or permissions:
   - CRD fields → `api/v1alpha1/*_types.go`
   - RBAC permissions → `// +kubebuilder:rbac` markers in `internal/controller/doc.go`
   Then run `make manifests` to flow them into the chart.

2. **The chart** — for anything about the install:
   - `charts/kscribe/values.yaml` — image, resources, persistence, LLM config, default policy
   - `charts/kscribe/templates/*` — Deployment, Service, PVC, ServiceAccount, ClusterRoleBinding, default DiagnosisPolicy

This split is inherent to operators: the CRD schema and RBAC are derived from the
Go code, so the Go markers own them. The chart owns deployment topology.

## Common tasks

**Add an RBAC permission**
1. Add/extend a `// +kubebuilder:rbac:...` marker in `internal/controller/doc.go`.
2. `make manifests` → updates `charts/kscribe/templates/role.yaml`.
3. `scripts/build-manifest.sh` → updates `deploy/kscribe.yaml`.
4. Commit all three (marker, role.yaml, deploy).

**Change a CRD field**
1. Edit the struct + markers in `api/v1alpha1/*_types.go`.
2. `make generate manifests` → updates deepcopy + `charts/kscribe/crds/*.yaml`.
3. `scripts/build-manifest.sh` → updates `deploy/kscribe.yaml`.

**Change deployment resources / env / image default**
1. Edit `charts/kscribe/values.yaml` (or the relevant template).
2. `scripts/build-manifest.sh` → updates `deploy/kscribe.yaml`.

## Verify

```sh
make manifests          # regenerate CRDs + RBAC into the chart
make manifest-check     # render deploy/kscribe.yaml and assert it is reproducible (no diff)
helm lint charts/kscribe
helm template kscribe charts/kscribe --include-crds --namespace kscribe-system
```

CI enforces this: the `test` job runs `make generate manifests templ && git diff
--exit-code` (generated files committed and current) and `make manifest-check`
(`deploy/kscribe.yaml` reproducible). The `e2e` job installs the chart into
minikube and checks the operator serves `/healthz`.
